# 6.1 檔案描述符與基本 I/O

## 什麼是檔案描述符？

在 Unix/Linux 中，**檔案描述符（File Descriptor, FD）** 是一個非負整數，代表一個已開啟的 I/O 資源。核心維護每個程序的 FD 表。

### 三個標準檔案描述符

| FD | 名稱 | 符號常數 | 一般連接 |
|----|------|---------|---------|
| 0 | 標準輸入 | `STDIN_FILENO` | 鍵盤 |
| 1 | 標準輸出 | `STDOUT_FILENO` | 螢幕 |
| 2 | 標準錯誤 | `STDERR_FILENO` | 螢幕 |

## 基礎 I/O 系統呼叫

```c
#include <unistd.h>   // read, write, close
#include <fcntl.h>    // open, O_RDONLY, etc.
#include <sys/stat.h> // stat
```

| 系統呼叫 | 功能 | 回傳值 |
|---------|------|--------|
| `open(path, flags, mode)` | 開啟檔案 | FD（成功）或 -1（失敗） |
| `read(fd, buf, count)` | 從 FD 讀取資料 | 實際讀取位元組數 |
| `write(fd, buf, count)` | 寫入資料到 FD | 實際寫入位元組數 |
| `lseek(fd, offset, whence)` | 移動讀寫位置 | 新位置偏移量 |
| `close(fd)` | 關閉檔案 | 0（成功）或 -1（失敗） |
| `dup(fd)` | 複製 FD | 新的 FD |
| `dup2(oldfd, newfd)` | 複製 FD 到指定值 | 成功回傳 newfd |

### 範例：讀取並複製檔案

```c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <fcntl.h>

#define BUFSIZE 4096

int main(int argc, char *argv[]) {
    if (argc != 3) {
        fprintf(stderr, "用法: %s <來源> <目的>\n", argv[0]);
        return 1;
    }

    int src = open(argv[1], O_RDONLY);
    if (src < 0) { perror("開啟來源失敗"); return 1; }

    int dst = open(argv[2], O_WRONLY | O_CREAT | O_TRUNC, 0644);
    if (dst < 0) { perror("開啟目的失敗"); close(src); return 1; }

    char buf[BUFSIZE];
    ssize_t n;
    while ((n = read(src, buf, BUFSIZE)) > 0) {
        write(dst, buf, n);
    }

    close(src);
    close(dst);
    printf("複製完成！\n");
    return 0;
}
```

**編譯執行：**

```bash
$ gcc -o filedemo filedemo.c
$ echo "Hello Systems Programming" > input.txt
$ ./filedemo input.txt output.txt
複製完成！
$ cat output.txt
Hello Systems Programming
```

## FD 與檔案結構

```
程序 A 的 FD 表                             核心
┌────────────┐                            ┌──────────────────┐
│ FD 0: stdin │────→ 開啟檔案描述 (File Description) ──→ inode
│ FD 1: stdout│────→ 開啟檔案描述 (File Description) ──→ inode
│ FD 2: stderr│────→ 開啟檔案描述
│ FD 3: (空)  │
│ ...         │
└────────────┘
```

多個 FD 可以指向同一個開啟檔案描述（經由 `dup()`），而一個檔案可以被多個程序開啟。

## 思考題

> **Q**：為什麼 `read()` 和 `write()` 的回傳值可能小於請求的位元組數？  
> **A**：對於某些檔案類型（如 socket、pipe），資料可能分批到達。read/write 回傳實際處理的位元組數，呼叫者必須在迴圈中處理剩餘資料，直到回傳值為 0（EOF）或 -1（錯誤）。
