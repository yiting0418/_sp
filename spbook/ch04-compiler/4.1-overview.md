# 4.1 編譯流程概觀

## 編譯器 vs 直譯器

| 特性 | 編譯器 (Compiler) | 直譯器 (Interpreter) |
|------|-----------------|-------------------|
| 執行方式 | 先翻譯整體，再執行 | 逐行翻譯並立即執行 |
| 執行速度 | 快（已最佳化機器碼） | 慢（每次重新解析） |
| 除錯便利性 | 較難 | 易（錯誤即時回報） |
| 跨平台 | 低（平台相依） | 高（直譯器可移植） |
| 代表語言 | C、C++、Rust、Go | Python、Ruby、Shell |

## GCC 的四個編譯階段

```
原始碼 (.c)
    │
    ▼ 前置處理（Preprocessing）
前置處理後的原始碼 (.i)    ← gcc -E
    │
    ▼ 編譯（Compilation）
組合語言 (.s)              ← gcc -S
    │
    ▼ 組譯（Assembly）
目的碼 (.o)               ← gcc -c
    │
    ▼ 連結（Linking）
執行檔 (a.out)            ← gcc
```

### 階段詳解

```bash
# 假設有 hello.c

# 階段 1：前置處理
# 展開 #include、#define、條件編譯
$ gcc -E hello.c -o hello.i
$ wc -l hello.c hello.i
     6 hello.c
   841 hello.i    # stdio.h 展開後暴增！

# 階段 2：編譯 → 組合語言
$ gcc -S hello.i -o hello.s && head -5 hello.s
	.file	"hello.c"
	.section	.rodata
.LC0:
	.string	"Hello, World!"
	.text

# 階段 3：組譯 → 目的碼
$ gcc -c hello.s -o hello.o

# 階段 4：連結 → 執行檔
$ gcc hello.o -o hello && ./hello
Hello, World!
```

## 編譯器內部架構

```
   原始碼
      │
      ▼ 詞法分析（Lexer）
  Token 串流
      │
      ▼ 語法分析（Parser）
  抽象語法樹（AST）
      │
      ▼ 語意分析（Semantic）
  已檢查的 AST + 符號表
      │
      ▼ 最佳化（Optimizer）
  中間碼（IR）
      │
      ▼ 程式碼生成（Code Gen）
  機器碼
```

## Java：半編譯半直譯

Java 採用獨特的混合模式：

1. **編譯階段**：`javac` 將 `.java` 原始碼編譯成 `.class` 位元組碼（Bytecode）
2. **直譯階段**：JVM 讀取 Bytecode 並直譯執行
3. **JIT 編譯**：對熱區程式碼（Hot Spot），即時編譯成機器碼提升效能

## 思考題

> **Q**：為什麼 C 程式比 Python 快許多？  
> **A**：C 編譯後直接是目標平台的機器碼，執行時 CPU 直接執行。Python 每次執行都要直譯（解析 + 執行 AST 或位元組碼），且有動態型別、GC 等執行期開銷。一般 C 比 Python 快 10~100 倍，視任務而定。
