# 2.3 競態條件與 Mutex 同步

## 競態條件（Race Condition）

當多個執行緒同時讀寫共享資料，且結果取決於執行緒的交錯順序（Timing）時，就會發生競態條件。

### 經典問題：counter++

`counter++` 看起來是一行，但編譯後是三步**非原子操作**：

```
步驟 1: 從記憶體讀取 counter 到暫存器  (LOAD)
步驟 2: 暫存器 + 1                     (INCREMENT)
步驟 3: 寫回記憶體                     (STORE)
```

若兩個執行緒同時執行：

```
時間 ▶   Thread 1              Thread 2         counter
        LOAD counter (=0)                        0
        INCREMENT (+1 → 1)                       0
                                LOAD counter (=0) 0    ← 讀到舊值！
                                INCREMENT (+1→1) 0
                                STORE counter=1  1    ← 覆蓋 T1 的結果！
        STORE counter=1         ← 实际上 T1 的值被覆盖了
                                             final: 1 (但期望是 2！)
```

## Mutex（互斥鎖）

Mutex 提供**互斥存取**——只有一個執行緒可以持有鎖並進入臨界區間。

### 有 Mutex 保護的正確範例

```c
#include <pthread.h>
#include <stdio.h>

int counter = 0;
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

void *increment(void *arg) {
    for (int i = 0; i < 100000; i++) {
        pthread_mutex_lock(&lock);   // 🔒 上鎖
        counter++;                   // 臨界區間（只能一個人進來）
        pthread_mutex_unlock(&lock); // 🔓 解鎖
    }
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, increment, NULL);
    pthread_create(&t2, NULL, increment, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);

    printf("有 Mutex 保護的結果: %d\n", counter);
    printf("期望值:               %d\n", 200000);
    printf("結果%s\n",
           counter == 200000 ? "✓ 正確！" : "✗ 發生競態條件！");
    return 0;
}
```

**執行結果：**

```
$ gcc -pthread -o mutex_demo mutex_demo.c && ./mutex_demo
有 Mutex 保護的結果: 200000
期望值:               200000
結果✓ 正確！
```

## 死結（Deadlock）

當兩個執行緒互相等待對方持有的鎖時，就會發生死結：

```
Thread A:  lock(m1) → lock(m2) → ... unlock...
Thread B:  lock(m2) → lock(m1) → ... unlock...
// 如果 A 持有 m1 同時 B 持有 m2，雙方永遠無法取得第二把鎖！
```

### 避免死結的原則

1. **固定鎖順序**：所有執行緒以相同順序取得鎖
2. **使用 trylock**：`pthread_mutex_trylock()` 避免阻塞
3. **限制鎖持有時間**：臨界區間越小越好

## 思考題

> **Q**：如果去掉 Mutex，counter 的最終值可能是多少？  
> **A**：小於 200000 的某個不確定數值。因為 counter++ 不是原子操作（讀取→加一→寫回共三步），兩個執行緒可能同時讀到相同的舊值，導致其中一次加法被覆蓋，稱為「更新遺失（Lost Update）」。
