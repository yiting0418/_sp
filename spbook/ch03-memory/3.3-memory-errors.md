# 3.3 常見記憶體錯誤

## 錯誤類型

### 1. 記憶體洩漏（Memory Leak）

忘記 `free()` 已配置的記憶體：

```c
// ❌ 錯誤
void leaky() {
    int *p = malloc(100);
    // 忘記 free(p)！
    // 程式結束前，這 100 位元組永遠無法被回收
}

// ✅ 正確
void clean() {
    int *p = malloc(100);
    // ... 使用 p ...
    free(p);
    p = NULL;
}
```

### 2. 緩衝區溢位（Buffer Overflow）

存取超出配置範圍的記憶體：

```c
// ❌ 錯誤
int arr[5];
arr[10] = 999;  // 寫入超出範圍！→ 可能覆蓋其他資料或造成 Segmentation Fault
```

### 3. 懸空指標（Dangling Pointer）

釋放記憶體後繼續使用該指標：

```c
// ❌ 錯誤
int *p = malloc(4);
free(p);
*p = 99;  // free 後繼續使用！→ 未定義行為（可能崩潰、資料損毀、或被利用攻擊）
```

### 4. 雙重釋放（Double Free）

對同一個指標呼叫 `free()` 兩次：

```c
// ❌ 錯誤
int *p = malloc(4);
free(p);
free(p);  // 雙重釋放！→ 未定義行為，可能導致 Heap 損毀
```

### 5. 未初始化的記憶體

```c
// ❌ 錯誤
int *p = malloc(5 * sizeof(int));
printf("%d\n", p[0]);  // 垃圾值！不確定的內容
```

## 正確示範

```c
#include <stdio.h>
#include <stdlib.h>

void demo_memory_errors() {
    // ✅ 正確做法示範
    int *safe = (int *)malloc(5 * sizeof(int));
    if (!safe) return;

    for (int i = 0; i < 5; i++) safe[i] = i;  // 只寫入合法範圍
    printf("正確存取: ");
    for (int i = 0; i < 5; i++) printf("%d ", safe[i]);
    printf("\n");

    free(safe);
    safe = NULL;  // free 後立即設 NULL，防止意外使用
    printf("safe == NULL: %s\n", safe == NULL ? "true ✓" : "false");
}

int main() {
    demo_memory_errors();
    return 0;
}
```

**執行結果：**

```
$ gcc -o memerr memerr.c && ./memerr
正確存取: 0 1 2 3 4
safe == NULL: true ✓

$ valgrind --leak-check=full ./memerr
==5120== All heap blocks were freed -- no leaks are possible
```

## 使用 Valgrind 檢查

Valgrind 是 Linux 上最強大的記憶體檢測工具：

```bash
# 基本記憶體洩漏檢查
valgrind --leak-check=full ./myprogram

# 追蹤未初始化變數
valgrind --track-origins=yes ./myprogram

# 檢測所有可能的問題
valgrind --tool=helgrind ./mythread_program  # 執行緒錯誤
```

## 最佳實踐總結

| 原則 | 說明 |
|------|------|
| **malloc 後檢查 NULL** | 確保配置成功 |
| **free 後設 NULL** | 防止懸空指標 |
| **初始化變數** | 不要假設記憶體自動歸零 |
| **越界檢查** | 確認 array index 在範圍內 |
| **誰配置誰釋放** | 明確的記憶體所有權 |
| **使用 Valgrind** | 定期檢查記憶體問題 |

## 思考題

> **Q**：為什麼 `free()` 後要將指標設為 `NULL`？  
> **A**：如果不設為 NULL，該指標仍然指向已釋放的記憶體（懸空指標）。後續若意外再次使用此指標（讀取或寫入），會造成未定義行為。設為 NULL 後，若不小心使用，至少會立即產生 Segmentation Fault，讓問題在發生當下就被發現，而不是隱藏成難以追蹤的 Bug。
