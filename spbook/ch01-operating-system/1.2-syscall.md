# 1.2 使用者模式 vs 核心模式與系統呼叫

## 兩種執行模式

現代處理器（如 x86、ARM）提供至少兩種執行模式，防止使用者程式破壞系統：

| 特性 | 使用者模式 | 核心模式 |
|------|-----------|---------|
| 執行程式 | 應用程式 | OS 核心 |
| 硬體存取 | ❌ 禁止 | ✅ 直接存取 |
| 特權指令 | ❌ 無法執行 | ✅ 可執行 |
| 記憶體範圍 | 僅自己的位址空間 | 整個實體記憶體 |
| 模式切換 | 系統呼叫、中斷時觸發 | 開機時進入 |

## 系統呼叫（System Call）

當使用者程式需要 OS 服務時（如讀檔、發送網路封包），必須透過系統呼叫請求 OS 代為執行。

### 系統呼叫流程

```
Userspace:  write(fd, buf, len)
              │
              ▼   CPU 切換到核心模式
Kernelspace: sys_write()
              │
              ▼
            裝置驅動程式 → 硬體
              │
              ▼   CPU 切回使用者模式
Userspace:  返回結果
```

### 程式範例：使用系統呼叫

```c
#include <unistd.h>
#include <fcntl.h>
#include <stdio.h>
#include <string.h>

int main() {
    // write() 是系統呼叫，請求 OS 輸出到標準輸出
    const char *msg = "Hello from user space!\n";
    write(STDOUT_FILENO, msg, strlen(msg));

    // getpid() 取得目前程序 ID（也是系統呼叫）
    pid_t pid = getpid();
    printf("目前程序 PID: %d\n", pid);
    printf("父程序 PID:   %d\n", getppid());

    return 0;
}
```

**編譯執行：**

```bash
$ gcc -o syscall_demo syscall_demo.c && ./syscall_demo
Hello from user space!
目前程序 PID: 4821
父程序 PID:   1534
```

### 常見系統呼叫

| 系統呼叫 | 用途 | 對應 C 函式 |
|----------|------|-------------|
| `read()` | 從檔案描述符讀取 | `fread()` |
| `write()` | 寫入檔案描述符 | `fwrite()` |
| `open()` | 開啟檔案 | `fopen()` |
| `close()` | 關閉檔案 | `fclose()` |
| `fork()` | 複製程序 | — |
| `execve()` | 載入新程式 | `execvp()` |
| `mmap()` | 映射記憶體 | — |
| `sched_yield()` | 自願讓出 CPU | — |

### 為什麼系統呼叫很慢？

系統呼叫比一般函式呼叫慢 **10~100 倍**，原因：

1. **模式切換（Mode Switch）**：CPU 必須從使用者模式切換到核心模式
2. **上下文保存**：必須保存暫存器、堆疊等狀態
3. **權限檢查**：核心必須驗證請求的合法性
4. **快取污染**：切換可能導致 CPU 快取失效

> **提示**：`strace` 工具可以追蹤程式的所有系統呼叫：
> ```bash
> strace -c ./syscall_demo
> ```

## 思考題

> **Q**：系統呼叫與一般函式呼叫有什麼本質差異？  
> **A**：一般函式呼叫在使用者空間執行；系統呼叫會觸發 CPU 從使用者模式切換到核心模式，執行完後再切回。這個模式切換有一定的效能開銷，因此系統呼叫比一般函式呼叫慢許多。
